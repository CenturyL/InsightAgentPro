## 1. 项目定位

这个项目不是一个“单纯能聊天的 API”，而是一个**可扩展的 Agent 后端平台**。

它的目标不是只完成一次问答，而是同时解决以下几类工程问题：

- 如何让 Agent 在简单问题和复杂任务之间自动切换
- 如何让知识库检索更像真实业务系统，而不是只做最基础的向量相似度查询
- 如何让用户的长期偏好和历史事实能够跨会话保留
- 如何对 RAG 和 Agent 的效果做量化评估
- 如何快速准备一套可复现实验环境，方便开发、演示和学习

所以这个项目的本质是：

**一个支持 ReAct、Plan-and-Execute、现代 RAG、长期记忆和评估闭环的 Agent 服务底座。**

补充一点：

- 系统仍然保留公司内部知识查询、办公制度查询和时间查询等辅助能力
- 但这些能力已经从默认主设定降级为辅助 tool
- 系统的主身份、人设和主要场景统一收敛为政策通知与招投标分析

## 2. 解决了什么问题

传统的本地问答项目通常存在这些问题：

1. 只有一个聊天接口，没有任务类型区分
2. 只有向量检索，没有重排和过滤
3. 没有长期记忆，换个线程就“失忆”
4. 没有评估体系，只能靠主观感觉说“效果还行”
5. 没有测试数据准备流程，换台机器就难以复现

本项目当前版本针对这些点分别做了处理：

- 用 `task_mode + complexity routing` 区分简单问题和复杂任务
- 用 `dense + lexical + rerank + metadata filter` 构成更完整的 RAG 主链路
- 用 `LangGraph checkpointer + pgvector` 实现短期/长期双轨记忆
- 用 `retrieval_eval + generation_eval` 跑离线指标
- 用 `rebuild_test_environment` 一键重建测试文档、评估集和入库环境

```text
        │

FastAPI API Layer
  - /chat/stream
  - /chat/agent
  - /knowledge/upload
  - /eval/retrieval
  - /eval/generation
  - /testing/rebuild
        │
        ▼
Service Layer
  - agent_service
  - rag_service
  - eval_service
  - test_env_service
        │
        ▼
Agent Orchestration Layer
  - ReAct path
  - Plan-and-Execute path
  - middleware
  - LangGraph orchestrator
        │
        ▼
Retrieval / Memory Layer
  - retrieval.pipeline
  - retrieval.citation
  - pgvector long-term memory
  - LangGraph short-term memory
        │
        ▼
Model Layer
  - ChatOllama (Qwen)
  - ChatOpenAI compatible (DeepSeek)
  - Embedding model
  - Reranker model
        │
        ▼
Data Layer
  - Chroma (当前知识库)
  - PostgreSQL + pgvector
  - 本地评估数据集
  - 测试文档
```

### 3.2 为什么分层

这样分层不是为了“看起来高级”，而是为了让每层只承担一类责任：

- API 层只负责接请求和回响应
- Service 层负责对外暴露业务动作
- Agent 层负责推理和编排
- Retrieval 层负责知识检索
- Memory 层负责状态与记忆
- Data 层负责落地存储

这意味着以后你改检索算法，不需要重写路由；你改模型，也不需要重写评估接口。

## 5. 核心请求链路

### 5.1 `/chat/agent` 的请求路径

一个典型请求的流转如下：

1. FastAPI 在 `api/routes.py` 接收到 `ChatRequest`
2. 路由层把 `query / thread_id / user_id / task_mode / metadata_filters` 交给 `get_agent_stream`
3. `agent_service.py` 判断：
   - 如果是简单问题，走现有 `create_agent + middleware` 路径
   - 如果是复杂任务，走 `LangGraph Plan-and-Execute` 路径
4. 检索或工具执行时，会调用 `retrieval.pipeline` 或 `tools.py`
5. 若开启长期记忆，middleware 会在模型调用前把相关记忆注入 prompt
6. 最终结果流式输出给前端
7. 对话结束后，后台任务会尝试抽取稳定用户信息写入长期记忆

### 5.2 `/eval/*` 的请求路径

#### `/eval/retrieval`

1. FastAPI 接收 `RetrievalEvalRequest`
2. `eval_service.py` 调 `retrieval_eval.py`
3. 读取 JSONL 数据集
4. 逐条调用 `retrieve_knowledge_bundle`
5. 计算 Precision@k、Recall@k、MRR、nDCG、hit_rate
6. 返回汇总结果和 per-query 详情

#### `/eval/generation`

1. FastAPI 接收 `GenerationEvalRequest`
2. `eval_service.py` 调 `generation_eval.py`
3. 逐条样本：
   - 先检索证据
   - 再跑真实 agent 生成答案
   - 再调用 judge 模型打分
4. 汇总 answer_relevance、faithfulness、citation_accuracy、keyword_coverage

## 6. 核心模块详解

## 6.1 配置层：`core/config.py`

这里统一管理：

- Ollama base URL
- 本地模型名
- DeepSeek API Key / base URL / model
- 向量库路径
- Embedding / Reranker 模型名
- PostgreSQL 连接串

设计原则：

- 所有环境差异都通过 `.env` 管理
- 不把模型地址、数据库地址散落到业务代码里

---

## 6.2 模型层：`core/llm.py`

当前项目有两个核心模型对象：

- `basic_model`：本地 Qwen，适合低成本、低延迟、简单任务
- `advanced_model`：DeepSeek，适合规划、反思、复杂总结和评估

为什么要双模型：

- 简单问题没必要用高成本强模型
- 复杂任务如果全交给本地模型，稳定性和规划能力通常不够

因此系统不是“固定用一个模型”，而是做了动态路由。

---

## 6.3 Middleware 层：`core/middleware.py`

当前 middleware 做了三件事：

### 1. 长期记忆注入

在每次模型调用前，根据 `user_id + 当前 query` 去 pgvector 检索相关长期记忆，然后拼到 `system_prompt` 里。

### 2. 动态模型选择

根据 query 长度、对话轮次和关键词，把请求分到本地模型或 DeepSeek。

### 3. 工具异常兜底

如果工具执行失败，不让整个 Agent 崩掉，而是返回一个可被模型继续处理的 `ToolMessage`。

这是非常典型的“横切关注点”设计：

- 记忆注入不属于具体工具逻辑
- 模型路由不属于具体业务逻辑
- 异常兜底不属于具体检索逻辑

所以把它们抽到 middleware 里是合理的。

---

## 6.4 Agent 编排层：`agents/`

这是项目从“普通 tool-calling agent”升级到“任务型 Agent”的关键。

### `state.py`

定义 LangGraph 的执行状态，包括：

- messages
- user_id
- thread_id
- task_mode
- metadata_filters
- plan
- step_results
- citations
- final_answer

### `router.py`

负责判定当前请求是不是复杂任务。

如果命中以下特征，会优先进入复杂路径：

- query 很长
- 出现比较、分析、抽取、报告、表格、风险等关键词
- 前端显式指定 `task_mode=compare/extract/report`

### `planner.py`

负责把复杂问题拆成结构化计划。

当前设计不是直接相信模型的纯 JSON 输出，而是：

1. 先要求模型输出 JSON 数组
2. 再提取代码块里的 JSON
3. 再用 `pydantic TypeAdapter` 做结构校验
4. 如果失败，就回退到默认计划

这比直接 `json.loads()` 稳得多。

### `executor.py`

按步骤执行计划，核心是：

- 读取当前 step
- 调检索链路
- 收集证据
- 写入 step_results
- 聚合 citations

### `reflection.py`

当某一步失败或证据不足时，尝试扩大召回范围并补一轮检索。

作用：

- 提高复杂任务的鲁棒性
- 降低“一次召回失败就整题失败”的概率

### `orchestrator.py`

这里用 `StateGraph` 把节点串起来，当前图是：

```text
START
  → complexity_router
      → react_agent
      → planner
          → executor
              → reflection
                  → synthesizer
                      → END
```

`synthesizer` 还会根据 `task_mode` 决定输出结构：

- `compare`
- `extract`
- `report`
- `qa`

## 6.5 检索层：`retrieval/pipeline.py`

这是当前 RAG 主链路。

### 6.5.1 现在做了什么

当前 retrieval pipeline 具备：

- 向量召回
- 关键词补充召回
- merge 去重
- reranker 精排
- metadata filter
- citation 构造
- `RetrievalBundle` 标准输出

### 6.5.2 为什么不是只做向量检索

只做向量检索容易有三个问题：

1. 精确关键词不稳定
2. 时间、地区、source 等约束难以控制
3. 最终返回的上下文和引用不好统一封装

所以这里把检索封成了统一的 `retrieve_knowledge_bundle()`，输出：

- `docs`
- `context_text`
- `citations`
- `applied_filters`

这样调用方不用每次自己重新拼 prompt 和证据。

### 6.5.3 Hybrid Retrieval 的当前实现

当前版本的 hybrid 还不是“真正生产级 BM25 + 向量检索双索引”，而是一个过渡版：

- `dense_search_knowledge()` 负责向量召回
- `lexical_search_knowledge()` 负责关键词重打分
- 合并后再用 reranker 精排

这已经能体现现代 RAG 的思路，而且接口上为以后接 pgvector + PostgreSQL FTS 留了口子。

---

## 6.6 引用层：`retrieval/citation.py`

引用模块负责把文档 metadata 标准化成前端和评估都能消费的格式。

当前 citation 包含：

- source
- page
- section
- preview

为什么单独抽出来：

- 生成答案时需要引用
- 评估时需要 source 命中比对
- 前端渲染也可能需要点击查看证据

如果 citation 不统一，后面每层都会自己拼一遍，最终越来越乱。

---

## 6.7 记忆层：`core/memory.py` + `services/agent_service.py`

### 6.7.1 短期记忆

短期记忆由 LangGraph checkpointer 负责。

当前策略是：

- 有 `POSTGRES_URL` 就用 `AsyncPostgresSaver`
- 没有就退回 `MemorySaver`

短期记忆主要解决：

- 同一个 thread_id 的多轮上下文延续
- Agent 执行状态的持久化

### 6.7.2 长期记忆

长期记忆使用 pgvector。

核心逻辑：

- 对话结束后，抽取用户稳定信息
- 保存到 PGVector
- 下次根据 query 相似度检索注入

适合保存：

- 用户偏好
- 关注领域
- 固定输出习惯

不适合保存：

- 一次性聊天内容
- 每轮临时上下文

## 7. ReAct 与 Plan-and-Execute 如何并存

很多人做 Agent 项目时会陷入两个极端：

- 要么全是简单 tool calling，没有复杂任务能力
- 要么一上来所有请求都走 planner，导致延迟和复杂度爆炸

当前项目的做法是折中：

### 简单任务

走 ReAct：

- 响应快
- tool-calling 足够
- 适合日常问答和简单检索

### 复杂任务

走 Plan-and-Execute：

- 有明确计划
- 有步骤执行
- 有失败反思
- 更适合比较、抽取、报告型任务

这也是你在面试里非常值得讲的一点：

**不是“我会 Plan-and-Execute”，而是“我知道什么时候该用，什么时候不该用”。**

## 8. RAG 检索链路详解

一个完整的检索调用大致如下：

1. 用户 query 进入 pipeline
2. 系统尝试推断 metadata_filters
3. 先做 dense retrieval
4. 再做 lexical retrieval
5. 合并结果去重
6. 用 reranker 精排
7. 生成 citations
8. 返回 RetrievalBundle

### 当前已支持的 filter

- `source`
- `source_type`
- `region`
- `year`

### 为什么 filter 很重要

因为很多真实问题不是“语义相似就行”，而是带硬约束：

- 只看某个地区
- 只看某个年份
- 只看某一类来源
- 只看某个指定文档

没有 filter，RAG 很容易检索“差不多对，但不是你要的那份”。

## 9. 评估系统详解

## 9.1 检索评估：`evaluation/retrieval_eval.py`

当前支持的指标：

- Precision@k
- Recall@k
- MRR
- nDCG@k
- hit_rate

### 评估口径

这里按“source 级”评估，而不是按 chunk 级评估。

原因：

- 同一文档会被切成多个 chunk
- 如果按 chunk 算，很容易重复加分
- 最终会出现 `Recall > 1` 这种明显不合理的结果

所以现在先做 source 级口径，更适合作为简历项目的解释和演示。

## 9.2 生成评估：`evaluation/generation_eval.py`

当前生成评估流程：

1. 从数据集读取 query
2. 先跑 retrieval，拿到证据
3. 再跑真实 agent 生成答案
4. 清洗流式日志污染
5. 用 judge 模型打分
6. 汇总结果

当前指标：

- answer_relevance
- faithfulness
- citation_accuracy
- keyword_coverage

### 为什么要清洗流式日志

因为 `get_agent_stream()` 不只是返回最终答案，也会返回过程日志，比如：

- 任务路由
- 工具开始
- 工具结束

这些内容如果直接进入 judge，会污染生成评估。所以这里单独做了 `_sanitize_agent_answer()`。

## 10. 一键测试环境重建机制

这是当前版本一个非常实用的工程能力。

### 10.1 为什么需要它

如果每次换机器、换环境、换数据库都要手动做这些事：

- 找测试文档
- 下载文档
- 清洗
- 入库
- 重写评估集
- 跑评估

那这个项目就很难复现，也很难演示。

### 10.2 当前支持两种入口

#### 1. 命令行脚本

文件：

[rebuild_test_env.py](/Users/century/code/agent/langchainPro/local_agent_api/scripts/rebuild_test_env.py)

示例：

```bash
conda run -n agent python /Users/century/code/agent/langchainPro/local_agent_api/scripts/rebuild_test_env.py
```

可选参数：

```bash
--force-download
--skip-retrieval-eval
```

#### 2. API

接口：

`POST /api/v1/testing/rebuild`

请求体：

```json
{
  "force_download": false,
  "run_retrieval_eval": true
}
```

### 10.3 它具体做了什么

1. 下载公开文档
2. 把原始 HTML 转纯文本
3. 入库到 Chroma
4. 重写 retrieval/generation 评估数据集
5. 可选跑一次检索评估并返回结果

这非常适合：

- 你本地重置环境后快速恢复
- 面试前快速演示
- 以后把测试集升级成更多文档时持续复用

## 11. 当前测试数据与验证结果

### 11.1 当前默认主题

现在默认测试语料已经从通用技术文档切换成了更贴近项目主题的：

- 产业政策通知
- 奖补/专项资金申报通知
- 中国政府采购网公开招标公告
- 招标文件与附件类页面

当前一键重建测试环境后，实际保留下来的有效主题语料为：

- `10` 份有效主题文档
- `125` 个主题 chunk
- 其中 `policy = 44`、`tender = 81`
- 加上项目原有规则文档后，当前 Chroma 中共 `127` 个 chunk、`11` 个 source

这里有一个真实工程现象也值得记住：

- 新增抓取的 `3` 个 CCGP 页面实际返回了重复错误页
- 系统现在会自动识别这类“内容重复但 source 不同”的无效文档
- 并将其记入 `failed_sources`，避免污染评估集

### 11.2 当前评估集结构

当前默认会生成两套 retrieval 数据集：

- `retrieval_eval_dataset_smoke.jsonl`
  - 用于验证链路是否通
  - 采用精确 source 过滤
- `retrieval_compare_dataset.jsonl`
  - 用于 baseline 对比
  - 只保留弱 metadata filter，例如 `doc_category / region`
  - 增加了多文档相关 query

其中：

- smoke 集当前是 `10` 条 query
- compare 集当前是 `14` 条 query

### 11.3 smoke 检索结果

当前 `hybrid_rerank` 在 smoke 集上的结果是：

- `Precision@3 = 1.0`
- `Recall@3 = 1.0`
- `MRR = 1.0`
- `nDCG@3 = 1.0`
- `hit_rate = 1.0`

## 12. 技术选型理由

### 为什么是 FastAPI

- 原生异步
- 和 LangChain/LangGraph 的 async 流很契合
- 很适合流式接口

### 为什么是 LangChain + LangGraph

- LangChain 适合做 tool / model / retriever 组装
- LangGraph 适合做状态图、复杂任务编排、检查点

### 为什么当前知识库还是 Chroma

- 本地开发快
- 容易落地测试
- 现阶段重点先是把链路打通

### 为什么长期记忆用 pgvector

- 更适合持久化
- 更像真实生产系统
- 支持按用户隔离的语义检索

### 为什么评估先用 LLM-as-judge

- 开发速度快
- 不需要一上来就接完整复杂评估框架
- 足够支撑学习和简历展示

## 13. 当前项目的不足与下一步演进

当前项目虽然已经比最初版本完整很多，但还有明显升级空间：

### 1. 知识库主存储仍未完全迁移到 pgvector

现在是：

- 短期记忆 / 长期记忆 -> PostgreSQL
- 知识库检索 -> Chroma

下一步可以统一收敛到 PostgreSQL + pgvector。

### 2. 多模态已经起步，但还不够深

当前已经支持：

- `html` 结构化清洗入库
- `csv/tsv` 表格文本化入库
- `image` 通过 `tesseract` OCR 入库

但距离完整多模态 RAG 还有差距，下一步仍需要：

- 扫描 PDF 的版面恢复
- PDF 表格抽取
- 图片/图表摘要
- 页码、附件名、table_id 级别的 citation

### 3. generation_eval 仍然偏重

目前 judge 需要调用强模型，单样本延迟不低。

下一步可以加：

- timeout
- fast judge mode
- 批量异步评估

### 4. 检索数据集仍偏小

虽然现在已经比最初版本强很多，但本质上仍然是：

- `10` 份有效主题文档
- `14` 条 harder compare query

这足够支撑学习、调参和简历展示，但还不够支撑严肃 benchmark。

后续更值得扩展到：

- 政策 PDF
- 招标公告正文 + 附件
- 表格抽取样本
- 多文档对比与时间线样本
- OCR 噪声样本

## 14. 简历和面试怎么讲

### 简历写法建议

你可以把这个项目概括为：

**基于 FastAPI、LangChain 和 LangGraph 构建的多模式 Agent 后端平台，支持 ReAct 与 Plan-and-Execute 双模式推理、现代化 RAG、多模态知识接入、pgvector 长期记忆以及离线检索/生成评估。**

### 简历项目经历示例

**多模式 Agent 智能体后端平台**  
**技术栈：** FastAPI | LangChain | PostgreSQL | Chroma | BGE Reranker | RAG Eval

- 设计并实现 `ReAct + Plan-and-Execute` 混合 Agent 架构，简单请求走本地 Qwen Tool Calling Agent，复杂任务通过 `Planner -> Executor -> Reflection -> Synthesizer` 分步执行；结合 EasyTier 打通本机服务与异机 Windows 本地模型部署，并接入 DeepSeek API，实现本地模型与云端模型混合路由，兼顾隐私保护与复杂任务效果。  
- 构建统一 Retrieval Pipeline，整合向量召回、关键词补充召回、Cross-Encoder 重排序、Metadata Filter 与 Citation 输出，并实现 `html/csv/image OCR` 多模态入库；基于政策/招投标主题语料完成 **10 份有效文档、125 个主题 chunk** 入库，在 smoke 集上 **Precision@3、Recall@3、MRR、nDCG@3 均为 1.0**。  
- 搭建离线 baseline 对比与系统 benchmark，当前 compare 集上 `hybrid_rerank` 相比 `dense_only` 实现 **Recall@3 +3.57pp、MRR +7.15pp**，平均检索时延增加 **516.65ms**；系统端到端 benchmark 中，**平均检索时延 1518.38ms，复杂任务时延 40565.33ms，峰值内存 141.53MB**。

### 面试重点讲什么

建议重点讲四件事：

1. 你为什么要做双路径：ReAct + Plan-and-Execute
2. 你为什么把 retrieval 抽成统一 pipeline
3. 你为什么要做评估，而不是只做 demo
4. 你如何让测试环境一键重建，保证项目可复现
5. 你如何做 baseline 对比，并量化 rerank 的收益与代价

### 面试官很可能追问的问题

#### 1. 为什么不所有请求都走 Planner？

因为复杂路径延迟更高、成本更高，而且简单问题没必要拆计划。

#### 2. 你说是现代 RAG，和普通向量检索差在哪？

普通向量检索只有“找相似文本”，这里增加了：

- lexical 补充召回
- metadata filter
- rerank
- citation 标准化
- 评估闭环

#### 3. 你怎么证明它真的有效？

不是只靠口头说，而是有：

- retrieval_eval
- generation_eval
- retrieval_compare baseline 对比
- system benchmark
- 可重建的测试文档和评估数据集

## 15. 总结

这个项目当前已经从“基础 LangChain Demo”演进成了一个有明显工程结构的 Agent 平台原型。

它最值得学习和复盘的地方，不是某一个 API 调通了，而是下面这几个设计同时成立：

- Agent 不是单路径，而是 ReAct 与 Plan-and-Execute 并存
- 检索不是散落的函数，而是统一的 retrieval pipeline

## 16. 按文件逐个讲代码

这一章不再从“架构视角”讲，而是按你打开代码时最容易接触到的文件顺序来讲。

建议阅读顺序也是这一章的顺序：

1. 先看入口
2. 再看 API
3. 再看 service
4. 再看 agent / retrieval / evaluation
5. 最后回看 core 层

### 16.1 `main.py`

这是整个应用的启动入口。

你看它时要抓三件事：

- FastAPI 实例在哪里创建
- 路由是怎么挂载的
- 生命周期资源是怎么初始化的

这个文件的关键价值不在“功能多”，而在于它把整个系统串起来了。

最重要的点：

- `lifespan()` 启动时会调用 `agent_service.initialize()`
- 关闭时会调用 `agent_service.cleanup()`
- `app.include_router(chat_router, prefix="/api/v1")` 把 API 层挂进来

你可以把 `main.py` 理解成“总装配厂”。

### 16.2 `api/schemas.py`

这是 API 的数据入口定义。

当前最重要的几个 schema：

- `ChatRequest`
- `RetrievalEvalRequest`
- `GenerationEvalRequest`
- `RebuildTestEnvRequest`

学习时要注意：

- 这里定义的是“接口契约”
- 不是业务逻辑
- 一旦 schema 变了，前端和后端都会受影响

例如：

- `task_mode` 决定是普通问答还是 compare/extract/report
- `metadata_filters` 允许调用方显式指定 source、region、year 等过滤条件

你以后调接口时，优先看这里，而不是先看 service。

### 16.3 `api/routes.py`

这是 HTTP 路由分发层。

它做的事应该很克制：

- 收参数
- 调 service
- 包装返回值

当前你可以重点看这些路由：

- `/chat/stream`
- `/chat/agent`
- `/knowledge/upload`
- `/eval/retrieval`
- `/eval/generation`
- `/testing/rebuild`

读这个文件时要问自己：

- 这个接口调用的是哪个 service
- 返回的是同步结果还是流式结果
- 哪些异常在路由层被转换成了 HTTPException

如果你以后要加新接口，应该优先模仿这里的风格。

### 16.4 `services/agent_service.py`

这是当前项目最核心的 service。

它负责：

- 初始化 Agent 单例
- 初始化 checkpointer
- 暴露流式调用入口
- 分流 ReAct 和 Plan-and-Execute
- 处理长期记忆后台回写

你阅读这个文件时，建议按下面顺序看：

#### 先看 `_build_agent()`

它定义了当前简单路径下的 Agent 是怎么构建的：

- 用哪个模型
- 挂哪些 tools
- 挂哪些 middleware
- 用什么 checkpointer

这决定了“基础 agent”是什么。

#### 再看 `initialize()`

这里的关键是：

- 如果有 `POSTGRES_URL`，就启用 `AsyncPostgresSaver`
- 如果没有，就退回 `MemorySaver`

这体现的是“优雅降级”的工程思路。

#### 再看 `get_agent_stream()`

这是最值得理解的函数，因为它是用户问题进入系统后的主要入口。

逻辑是：

1. 判断是否复杂任务
2. 如果复杂，走 `_run_plan_and_execute()`
3. 如果简单，走原有 `agent.astream_events()`
4. 流式把 token、tool start、tool end 推给前端
5. 最后异步回写长期记忆

这个函数其实就是“系统主控室”。

### 16.5 `services/rag_service.py`

这个文件现在更像“兼容层 + RAG 使用层”。

因为真正的检索主逻辑已经被抽到 `retrieval/pipeline.py` 里了。

它当前做的事主要是：

- 对外保留 `process_and_store_document`
- 对外保留 RAG 链式调用入口
- 使用 retrieval bundle 构造 prompt 和 citations

你读它时要意识到：

- 它不是知识库底层
- 它更像“RAG 业务适配器”

### 16.6 `services/eval_service.py`

这个文件很薄，但很重要。

它的作用不是实现评估逻辑，而是把评估逻辑包装成对外可调用的 service。

你可以把它理解成：

- API 层和 evaluation 层之间的桥

当前主要暴露：

- `run_retrieval_eval_job()`
- `run_generation_eval_job()`

如果以后你要接 Celery、定时任务、批量评估，这里会是很自然的扩展点。

### 16.7 `services/test_env_service.py`

这是“一键重建测试环境”的核心。

建议你重点理解它为什么有价值：

- 不是因为代码复杂
- 而是因为它让项目变得可复现

这个文件负责：

1. 下载公开文档
2. 将 HTML 转成 clean text
3. 入库
4. 重写评估集
5. 可选跑检索评估

也就是说，它不是“测试辅助脚本”，而是“把测试依赖标准化”的基础设施。

### 16.8 `scripts/rebuild_test_env.py`

这个脚本是 `test_env_service.py` 的命令行入口。

它做的事情很简单：

- 解析命令行参数
- 调 `rebuild_test_environment()`
- 打印结果 JSON

为什么要单独有这个脚本：

- 因为不是所有测试都适合走 API
- 有时你只想在终端里一键重建环境

所以这个脚本非常适合：

- 本地开发
- 面试前演示
- 环境重置后快速恢复

### 16.9 `agents/state.py`

这里定义的是 LangGraph 的状态结构。

你可以把它理解成“图执行过程中能携带的共享变量”。

比如：

- `plan`
- `step_results`
- `citations`
- `final_answer`

为什么它重要：

- 因为每个 node 不是直接互相传函数参数
- 而是通过共享 state 协作

如果你看不懂 state，这个项目的复杂路径就很难真正吃透。

### 16.10 `agents/router.py`

这是复杂度路由器。

它不是大模型路由，而是“任务复杂度路由”。

作用是：

- 识别当前 query 是否应该走 Plan-and-Execute

你读这里时要理解一件事：

复杂任务不应该靠“拍脑袋”判断，而应该有可解释的规则。

当前规则包括：

- 文本长度
- 特定关键词

后面你还可以扩展：

- 是否涉及多实体
- 是否要求结构化输出
- 是否要求比较/报告

### 16.11 `agents/planner.py`

这个文件体现了“结构化规划”的核心。

学习时重点看两部分：

#### 1. Prompt 约束

它要求模型输出结构化步骤，而不是散文式计划。

#### 2. 解析与校验

这里不是直接 `json.loads()` 就完了，而是：

- 先从代码块提取 JSON
- 再做 `pydantic` 校验
- 再做 normalize
- 最后失败时 fallback

这说明你已经开始按“生产思维”在写 LLM 输出处理，而不是把模型输出当绝对可靠输入。

### 16.12 `agents/executor.py`

这个文件负责按步骤执行计划。

它当前的执行方式是：

- 逐步构造 step query
- 调统一 retrieval pipeline
- 收集证据和 citations
- 写入 `step_results`

这个文件的重点不是“复杂算法”，而是“把 plan 变成可落地动作”。

你可以把 planner 和 executor 理解成：

- planner 负责“想”
- executor 负责“做”

### 16.13 `agents/reflection.py`

Reflection 是复杂任务稳定性的关键。

如果某一步失败，系统不会马上放弃，而是：

- 扩大召回范围
- 改写 retry query
- 再补一次检索

这让系统从“一次失败就挂”变成了“失败后还能补救”。

这类设计非常适合写进面试回答，因为它体现了你对 Agent 鲁棒性的关注。

### 16.14 `agents/react_agent.py`

这个文件目前是一个占位型节点。

它的意义不是功能复杂，而是为了让 LangGraph 图里显式保留简单路径。

为什么要保留：

- 让复杂图有清晰分支
- 为以后把简单路径完全迁进图里留接口

### 16.15 `agents/orchestrator.py`

这是复杂任务图的主入口。

重点看两个东西：

#### 1. 图结构怎么连

这里定义了 StateGraph 的节点和边。

#### 2. `synthesizer_node()`

它决定最后如何根据：

- plan
- step_results
- citations
- task_mode

拼出最终答案。

这里尤其值得注意的是 `task_mode` 对输出格式的影响：

- compare 用对比结构
- extract 用字段结构
- report 用报告结构
- qa 用普通回答结构

这让同一个 Agent 后端能服务多种任务形态。

### 16.16 `retrieval/pipeline.py`

这是当前最值得反复看的文件之一。

因为它把以前散乱的 RAG 行为统一成了一条清晰主线。

当前主要函数：

- `get_vector_store()`
- `infer_metadata_filters()`
- `dense_search_knowledge()`
- `lexical_search_knowledge()`
- `search_knowledge()`
- `retrieve_knowledge_bundle()`
- `process_and_store_document()`

学习顺序建议：

1. 先看文档入库
2. 再看 dense/lexical 两路召回
3. 再看 merge + rerank
4. 最后看 bundle 输出

这里几乎就是整个知识检索系统的骨架。

### 16.17 `retrieval/citation.py`

这个文件很小，但它体现了“统一证据格式”的意识。

它把每条 doc 转成统一 citation：

- source
- page
- section
- preview

小文件不代表不重要。很多项目后期会混乱，就是因为这种基础格式没有统一抽象。

### 16.18 `evaluation/retrieval_eval.py`

这个文件建议你认真读，因为它直接决定“你怎么证明系统有效”。

重点看：

- 数据集读取
- source 归一化
- 去重后的 hit 计算
- MRR / nDCG 计算

这里还修过一个典型问题：

- 同一个 source 被多个 chunk 重复计分

这类问题如果你能在面试里主动讲出来，会很加分，因为这说明你不只是“会跑指标”，而是知道指标容易被什么噪音污染。

### 16.19 `evaluation/generation_eval.py`

这个文件是“生成评估”的主逻辑。

它做的不是简单字符串比对，而是：

1. 跑真实 agent
2. 获取模型答案
3. 清洗日志污染
4. 让 judge 模型打分
5. 输出细粒度评估明细

重点理解三个点：

- 为什么要有 `_sanitize_agent_answer()`
- 为什么要保留 `model_answer / retrieved_citations / evidence_preview`
- 为什么 judge 模型要和业务模型分开思考

### 16.20 `core/embedding.py`

这里管理 embedding 和 reranker 模型的单例。

为什么单独放：

- 模型初始化重
- 不适合每次检索都重新加载

它的核心价值是资源管理，而不是业务逻辑。

### 16.21 `core/memory.py`

这里是长期记忆管理器。

你重点看：

- PGVector 初始化
- save()
- search()

它的核心思想是：

- 把长期记忆当作“用户稳定事实的向量库”
- 而不是把整个聊天记录全塞进去

### 16.22 `core/middleware.py`

这是横切逻辑的集中地。

要读懂这个项目，必须读懂它。

重点看三个 middleware：

- `inject_long_term_memory`
- `dynamic_model_selection`
- `handle_tool_errors`

这三个东西分别解决：

- 记忆注入
- 模型路由
- 工具异常兜底

这是系统工程感最强的一层之一。

### 16.23 `core/llm.py`

这个文件很简单，但你要知道它的意义不在“代码量”，而在“统一模型出口”。

项目中任何地方需要强模型或轻模型，都应该从这里取，而不是到处 new 模型对象。

### 16.24 `core/config.py`

配置层的意义是把“环境依赖”从“业务逻辑”里抽出去。

你以后排查问题时，经常要先看这里：

- base_url 对不对
- model name 对不对
- POSTGRES_URL 对不对
- vector store path 对不对

### 16.25 `services/tools.py`

这是 Agent 的工具层。

当前工具虽然不多，但它体现了一个关键原则：

**把“模型不擅长但程序擅长”的能力抽成 tool。**

例如：

- 查当前时间
- 查知识库

以后你继续扩展时，任何第三方能力都应该优先考虑是不是该做成 tool，而不是硬塞进 prompt。

### 16.26 `requirements.txt`

这个文件不要只当依赖清单看。

学习时应该按模块理解它：

- Web/API：FastAPI、uvicorn、pydantic
- LangChain/LangGraph：agent 与图编排
- Retrieval：Chroma、HF embedding、reranker
- Memory：Postgres、pgvector、checkpointer

如果你能把依赖和架构层一一对应起来，说明你真的理解了这个项目，而不是只记住了几个包名。
