# Agent / RAG 面试高频问答

> 面向校招、实习和初中级大模型应用岗位的高频问题整理  
> 结合当前项目实现来回答，尽量做到“能落到代码和指标”

---

## 使用建议

这份文档的阅读方式不是背诵，而是分三层：

1. 先记住每个问题的短答版
2. 再理解“为什么这么答”
3. 最后把回答和你项目里的真实实现、真实指标对应起来

你在面试时要避免两种回答：

- 只讲概念，不讲你项目里怎么做
- 只讲“我用了某某框架”，不讲取舍和问题

## 1. 怎么处理幻觉？

### 短答版

我主要从三层处理幻觉：

- 检索层减少错证据
- 生成层要求基于证据回答
- 评估层用 faithfulness 和 citation accuracy 检查

### 可展开版

在我的项目里，幻觉不是靠一句“不要编造”解决的，而是分层做：

1. 检索层
- 用 `dense + lexical + rerank + metadata filter`
- 先尽量召回相关文档，再用 reranker 提高排序质量
- 减少模型看到错误上下文的概率

2. Prompt 层
- 明确要求“如果知识库没有答案，就直接说不知道”
- 要求尽量基于检索内容回答，而不是自由发挥

3. 输出层
- 返回 citation
- 让答案和证据绑定

4. 评估层
- 用 generation eval 评估 `faithfulness` 和 `citation_accuracy`
- 不是靠主观感觉判断有没有幻觉

如果面试官继续追问，你可以补一句：

> 本质上幻觉不能被完全消灭，只能通过“减少错误证据 + 强化约束 + 做离线评估”去降低。

---

## 2. 如果知识库里没有答案，你怎么处理？

### 短答版

优先拒答，而不是瞎编。

### 可展开版

我的处理原则是：

- 检索不到高质量证据时，明确返回“根据当前知识库无法提供准确答案”
- 不让模型把常识补全伪装成知识库答案

如果后续做得更完整，可以加：

- 检索分数阈值
- citation 数量阈值
- 低置信度回答模板

这比“无论如何给一个答案”更符合企业场景。

---

## 3. 如何处理用户意图模糊？

### 短答版

先做任务类型判断，必要时主动澄清，不急着直接回答。

### 可展开版

用户意图模糊常见有两种：

- 问题太短，比如“帮我看看这个”
- 目标不清，比如“分析一下政策”

我一般按三步处理：

1. 先做意图识别
- 是问答、比较、抽取还是报告

2. 如果还能勉强回答
- 先给一个保守结果
- 同时说明我默认了哪些前提

3. 如果问题会显著影响结果
- 主动追问澄清
- 比如“你是想比较申报条件、补贴金额，还是截止时间？”

后面如果你要加得更完整，可以做：

- Human-in-the-loop 澄清节点
- planner 在缺参时先提问再执行

## 4. Agent 和 workflow 的区别是什么？

### 短答版

workflow 是预定义流程，agent 是让模型参与决策。

### 可展开版

workflow 更像：

- 固定顺序
- 固定分支
- 每一步基本可预期

agent 更像：

- 模型自己决定下一步做什么
- 会根据上下文选择工具、规划步骤、修正过程

更准确地说：

- workflow 的控制权主要在代码里
- agent 的控制权一部分交给模型

我当前项目其实是两者结合：

- 简单链路偏 agent
- 复杂任务的图结构偏 workflow
- 但 planner/executor 里又让模型参与决策

所以更像“agentic workflow”，而不是纯 agent 或纯 workflow。

---

## 5. ReAct 和 Plan-and-Execute 的区别是什么？

### 短答版

ReAct 适合边想边做，Plan-and-Execute 适合复杂任务先规划再执行。

### 可展开版

ReAct 的特点：

- 链路短
- 实现简单
- 简单任务响应快
- 但复杂任务容易反复试错

Plan-and-Execute 的特点：

- 先出计划
- 再逐步执行
- 对复杂分析、比较、报告更稳
- 但时延更高

我项目里两者共存：

- 简单问答走 ReAct
- `compare / extract / report` 更倾向走 Plan-and-Execute

这也是为什么我的 benchmark 里复杂任务时延明显更高。

---

## 6. ReAct 一定比 Plan-and-Execute 更快吗？

### 短答版

通常更快，但不绝对。

### 可展开版

原因很简单：

- ReAct 少了显式 planner
- 通常推理轮次更少

但如果任务本身很复杂，ReAct 可能因为反复工具调用而变慢。  
相反，如果 Plan-and-Execute 的计划很短、步骤稳定，也不一定特别慢。

所以更准确的说法是：

- 简单任务通常 ReAct 更轻
- 复杂任务通常 Plan-and-Execute 更稳

---

## 7. 你怎么决定一个请求走 ReAct 还是 Plan-and-Execute？

### 短答版

按请求粒度判断，不是按用户固定绑定。

### 可展开版

我当前项目里是每次请求单独判断，主要依据有：

- query 长度
- 是否包含“比较、分析、抽取、报告、风险”等关键词
- 是否显式传了 `task_mode`

不是按 `thread_id` 固定走某条路径。

所以同一个用户：

- 上一轮可以走 ReAct
- 下一轮也可能因为问题更复杂而走 Plan-and-Execute

当前还没做执行中动态切换，但后面可以加：

- 连续工具失败后从 ReAct 升级到 planner
- 某个计划步骤过于简单时从 planner 降级到简单 agent

---

## 8. 复杂任务怎么拆解任务，才能更好调用工具？

### 短答版

先按目标拆，再按证据需求拆，最后按输出结构拆。

### 可展开版

我的拆解思路不是机械按句子拆，而是分三层：

1. 目标拆解
- 用户到底要比较、抽取还是生成报告

2. 证据拆解
- 每个子任务需要查什么资料
- 需要知识库、网页还是计算工具

3. 输出拆解
- 最终要表格、字段列表还是长报告

比如：

“比较上海和苏州政策差异”

可以拆成：

- 找上海政策
- 找苏州政策
- 抽关键字段
- 做差异对比
- 生成结构化总结

这样 planner 输出的不是模糊想法，而是可执行步骤。

---

## 9. 复杂任务执行失败了怎么办？

### 短答版

不要直接整轮失败，要有 reflection、重试和降级策略。

### 可展开版

我项目里现在已经有 reflection 节点，目的就是：

- 某步失败后不要直接崩
- 先看是检索问题、工具问题还是计划问题

常见处理方式：

- 重新检索
- 放宽 filter
- 换更简单的执行方式
- 返回部分结果并说明缺失项

如果系统想更稳，可以加：

- step 级 retry
- replanning
- tool fallback

---

## 10. 怎么让大模型结构化输出？

### 短答版

我会同时做 prompt 约束、schema 校验和失败兜底。

### 可展开版

只靠 prompt 让模型“请输出 JSON”通常不稳，所以要三层：

1. Prompt 约束
- 明确字段、类型和示例

2. 解析层
- 先提取代码块或 JSON 片段

3. 校验层
- 用 `pydantic` 或 `TypeAdapter` 做结构校验

4. 失败兜底
- 如果结构化失败，回退默认计划或默认字段

我项目里的 planner 就是这么做的，不是直接相信模型一定返回完美 JSON。

---

## 11. 如果模型输出格式经常不稳定，你会怎么优化？

### 短答版

先限制输出空间，再做程序兜底，必要时拆小任务。

### 可展开版

我一般会按优先级做：

- 减少字段数
- 把一个大 JSON 拆成多个小任务
- 给明确示例
- 用 schema 校验
- 失败后重试

如果再不稳，说明不是“格式问题”，而是任务本身太复杂，应该降低单次输出复杂度。

---

## 12. 超 token 怎么处理？

### 短答版

靠检索裁剪、chunk 控制、记忆筛选和上下文摘要。

### 可展开版

超 token 不能只靠“换更大上下文模型”，要从输入源头控制：

1. 检索层
- 不要把太多 chunk 塞给模型
- 只保留 top-k 且高相关的证据

2. chunk 层
- 单个 chunk 不宜过大
- 同时要防止切太碎

3. 记忆层
- 长期记忆只注入最相关的几条
- 不是把所有历史都放进 prompt

4. 对话层
- 长对话要做摘要
- 保留稳定事实，压缩冗余轮次

5. 生成层
- 对复杂任务先分步执行，而不是一次全喂给模型

---

## 13. 你的 chunk 是怎么切的？

### 短答版

我已经从固定字符切升级成了标题层级 + 段落语义窗口 + 表格窗口的混合切块。

### 可展开版

最开始只是固定字符切，这种方式能跑，但不够好。  
现在我的做法更接近结构化切块：

- 先识别标题层级，形成 parent block
- 文本块按段落做语义窗口切分
- 表格块按行窗口切分
- 每个 chunk 带 `section_path / block_type / doc_category / region / year` 等 metadata

这样做的好处是：

- 检索更接近文档原始结构
- 比较适合政策、招标公告、表格类文档
- 后面做 citation 也更好解释

---

## 14. 为什么不能只用固定字符切 chunk？

### 短答版

因为容易把一个语义单元切断，也容易把不相关内容硬拼在一起。

### 可展开版

固定字符切的问题主要是：

- 章节边界被破坏
- 表格被切烂
- 标题和正文断开
- 检索命中后上下文不完整

所以固定字符切更适合第一版 demo，不适合长期迭代的业务型 RAG。

---

## 15. 你为什么要做 rerank？

### 短答版

因为粗召回和最终排序不是一个问题。

### 可展开版

向量召回的作用是：

- 从大语料里快速找候选

但候选不等于最终最相关结果。  
rerank 的作用是：

- 对 query 和候选文档做更细粒度匹配
- 解决“都差不多相关，但谁更该排前面”的问题

我项目里的真实结果也说明它有价值：

- `hybrid_rerank` 相比 `dense_only`
- `Recall@3 +3.57pp`
- `MRR +7.15pp`

当然代价也有：

- 平均检索时延增加 `516.65ms`

这个 trade-off 是可以拿来讲的。

---

## 16. 你为什么还要做 lexical retrieval？

### 短答版

因为纯向量检索对精确术语、简称、编号类信息可能不稳。

### 可展开版

向量检索擅长语义相似，但对这些情况容易失手：

- 精确词
- 特定项目名
- 法条编号
- 缩写
- 采购条目名称

所以我补了一路 lexical retrieval：

- 用关键词匹配补召回
- 再和 dense 结果合并
- 最后再 rerank

这其实是一个很常见、也很实用的工业化思路。

---

## 17. metadata filter 的作用是什么？

### 短答版

它是把“检索相关”变成“业务相关”的关键一步。

### 可展开版

很多时候“语义相似”不等于“业务上正确”。  
比如用户问的是：

- 上海地区
- 2025 年
- 政策类文档

如果你不做 filter，就可能召回：

- 北京政策
- 招标公告
- 老年份文档

所以 metadata filter 的价值是：

- 先缩小检索空间
- 再做语义排序

这能显著减少“看起来很像但其实不是目标文档”的误召回。

## 18. 评估 RAG 你主要看哪些指标？

### 短答版

我分检索评估和生成评估两类。

### 可展开版

检索评估：

- `Precision@k`
- `Recall@k`
- `MRR`
- `nDCG@k`
- `hit_rate`

生成评估：

- `answer_relevance`
- `faithfulness`
- `citation_accuracy`
- `keyword_coverage`

另外我还加了：

- baseline 对比
- 检索时延
- 系统 benchmark

这样就不只是“答得对不对”，还能看“值不值得”。

## 19. Precision、Recall、MRR、nDCG 分别是什么意思？

### 短答版

- Precision 看取回来的有多少对
- Recall 看该取回来的取回来多少
- MRR 看第一个正确结果排得靠不靠前
- nDCG 看整体排序质量

### 可展开版

如果面试官问，你可以这样答：

- `Precision@k`：前 k 个结果里有多少是相关的
- `Recall@k`：所有相关结果里，有多少被前 k 个取回来了
- `MRR`：如果第一个正确结果排第 1 就很好，排第 3 就差一些
- `nDCG`：考虑整个排序列表的质量，不只看第一个命中

这四个指标结合起来，比只看“是否命中”更完整。

## 20. 你的 benchmark 口径是什么？

### 短答版

我现在明确区分了 smoke 验证集、harder compare 集和系统 benchmark。

### 可展开版

当前项目里的 benchmark 不是一个模糊概念，而是三种口径：

1. smoke 集
- 验证链路是否通
- source 过滤强

2. compare 集
- 用弱 metadata filter
- 用于 dense / rerank / hybrid 做 baseline 对比

3. system benchmark
- 跑真实 agent 链路
- 测检索时延、简单请求时延、复杂请求时延、峰值内存

所以我在说结果时会明确说明：

- 数据集大小
- filter 强度
- 是离线指标还是端到端指标

这样口径更严谨。

---

## 21. 为什么你现在的 smoke 指标是 1.0，不代表系统无敌？

### 短答版

因为 smoke 集本来就是链路验证集，不是困难 benchmark。

### 可展开版

smoke 集的作用是：

- 验证下载、清洗、入库、检索、citation、评估是否通

所以它通常：

- query 更直接
- source 更明确
- filter 更强

因此分数高是合理的，但不能拿它代表真实复杂场景。

真正看差异，要看 compare 集和 baseline 对比。

## 22. 长期记忆的机制是什么？

### 短答版

长期记忆不是存整段聊天，而是存用户稳定事实和偏好，并在模型调用前按相似度注入。

### 可展开版

我项目里的长期记忆机制是：

1. 存什么
- 用户偏好
- 稳定事实
- 长期关注主题

2. 不存什么
- 整段原始对话
- 大量无关上下文

3. 怎么用
- 每次模型调用前
- 用当前 query 去搜索该用户相关记忆
- 取 top-k 相关记忆注入 system prompt

4. 为什么这么做
- 避免 prompt 膨胀
- 避免把短期噪声当长期事实

这比“把聊天记录全拼进去”更工程化。

---

## 23. 短期记忆和长期记忆的区别是什么？

### 短答版

短期记忆保留当前会话上下文，长期记忆保留跨会话稳定信息。

### 可展开版

短期记忆：

- 以线程或会话为单位
- 解决“这一轮对话别失忆”
- 通常存在 checkpointer 或 message history 里

长期记忆：

- 以用户为单位
- 解决“下次再来还能记得你”
- 通常存稳定偏好和事实

所以它们解决的是两个完全不同的问题，不能混为一谈。

---

## 24. 你为什么不用整个聊天记录直接做长期记忆？

### 短答版

因为噪声太大、成本太高，而且会把短期信息误当长期事实。

### 可展开版

如果把整段对话都塞进长期记忆，会有三个问题：

- 信息冗余
- 用户临时表达被误存
- 检索时相关性变差

所以更好的方式是：

- 先抽取稳定事实
- 再向量化保存

这样长期记忆更干净，也更容易命中真正有价值的信息。

---

## 25. 怎么避免长期记忆把错误信息越记越多？

### 短答版

要控制写入条件，不能每轮都盲写。

### 可展开版

更稳妥的做法包括：

- 只写稳定事实
- 对记忆做去重
- 对相似记忆做合并
- 给记忆加时间戳和来源
- 必要时支持覆盖或失效机制

我现在项目里已经做了“只注入少量相关记忆”和“按用户隔离”，后面还可以继续加记忆治理。

---

## 26. 用户问题特别长、上下文特别乱，你怎么处理？

### 短答版

先压缩问题，再识别任务，再决定检索和执行策略。

### 可展开版

用户问题过长时，不适合直接原样扔给 planner 或 retriever。  
更稳的做法是：

- 提取核心目标
- 提取约束条件
- 去掉情绪化和重复表达
- 再决定走 ReAct 还是 Plan-and-Execute

如果真的很长，还可以先做一层 query rewrite 或 summary。

---

## 27. 工具很多时，怎么避免模型乱调工具？

### 短答版

靠工具分层、提示词约束和任务拆解。

### 可展开版

工具一多，模型会出现：

- 多余调用
- 错误调用
- 顺序混乱

应对方法：

- 工具职责单一
- 工具描述写清楚输入输出
- 对复杂任务先拆步骤
- 给 planner 指定每步可能用哪些工具
- 做 tool error handling

工具治理做得好，agent 才不会变成“随机工具调用器”。

---

## 28. Tool calling 失败时你怎么处理？

### 短答版

不能让一个工具报错把整轮对话拖死，要做兜底。

### 可展开版

我项目里已经有 tool error middleware，核心思想是：

- 工具失败不等于整个系统失败
- 给模型一个可理解的错误反馈
- 让它换思路或跳过该步骤

常见处理方式：

- 返回可读错误信息
- 限制重试次数
- 触发 reflection
- 返回部分结果

---

## 29. 多模态 RAG 你现在做到什么程度了？

### 短答版

已经支持 `html/csv/image OCR` 入库，但还不是完整多模态 RAG。

### 可展开版

当前我已经做了：

- HTML 结构化清洗
- CSV/TSV 表格文本化
- 图片 OCR 入库

但还没完全做到的包括：

- 扫描 PDF 版面恢复
- PDF 表格抽取
- 图片/图表语义摘要
- 页码级和 table_id 级 citation

所以我会说：

> 当前已经具备轻量多模态入口，但仍处于多模态 RAG 的早期版本。

## 30. 你为什么说这个项目有工程价值，而不只是 demo？

### 短答版

因为它不只是“能对话”，而是有路由、检索、记忆、评估、benchmark 和可复现环境。

### 可展开版

一个 demo 通常只有：

- 一个聊天接口
- 一个向量检索
- 没有评估

而我这个项目已经有：

- ReAct 和 Plan-and-Execute 双路径
- 统一 retrieval pipeline
- 长期记忆
- baseline 对比
- system benchmark
- 一键重建测试环境

这说明它已经具备了“可分析、可优化、可复现”的工程特征。

## 31. 如果让你继续优化这个项目，你会先做什么？

### 短答版

我会先扩大主题数据集，再补更深的多模态解析，最后把知识库主存储统一到 pgvector。

### 可展开版

优先级我会这样排：

1. 扩评估数据
- 把 compare 集做到 30 到 50 条

2. 深化多模态
- 扫描 PDF OCR
- 表格抽取
- 附件级 citation

3. 优化执行链路
- 动态切换 ReAct 和 planner
- reflection + replanning

4. 存储统一
- 从 Chroma 进一步收敛到 PostgreSQL + pgvector

因为只有数据集先变强，后面的优化结果才更有说服力。

## 32. 如果面试官问“你最大的技术难点是什么”，你怎么答？

### 短答版

我会答“不是某个 API 调不通，而是如何把检索、规划、记忆和评估串成一套可量化迭代的系统”。

### 可展开版

我会重点讲两个难点：

1. 路由和编排
- 不是所有问题都适合 planner
- 需要在效果和时延之间平衡

2. 评估与指标
- 不只是做出功能
- 还要知道 dense、hybrid、rerank 哪个更值
- 要能量化收益和代价

这个回答通常比“我调了很多 prompt”更有工程感。

## 33. 如果面试官问“你学到了什么”，你怎么答？

### 短答版

我会答：Agent 项目最难的不是把模型接上，而是建立清晰边界、控制复杂度和做量化验证。

### 可展开版

我会总结三点：

- 模型不是万能控制器，复杂任务必须配合 workflow 和状态管理
- RAG 的重点不是“能搜”，而是“搜得准、解释得清、指标能验证”
- 真正能讲清楚的项目，必须有 benchmark、错误分析和复现能力

## 最后建议

如果你准备面试，不要把这 30 多个问题都背成书面答案。  
更好的方式是：

1. 每题只记一句短答版
2. 选 8 到 10 题重点练展开
3. 展开时一定带上你项目里的真实实现和真实数字

比如你现在最值得记住的几组数字：

- `10` 份有效主题文档
- `125` 个主题 chunk
- smoke 集 `Precision@3 / Recall@3 / MRR / nDCG@3 = 1.0`
- compare 集里 `hybrid_rerank` 相比 `dense_only`：
  - `Recall@3 +3.57pp`
  - `MRR +7.15pp`
  - 平均检索时延 `+516.65ms`
- system benchmark：
  - 平均检索时延 `1518.38ms`
  - 复杂任务时延 `40565.33ms`
  - 峰值内存 `141.53MB`

你一旦能把“设计 + 取舍 + 指标”三件事一起说出来，这个项目在面试里就不再像 demo。
